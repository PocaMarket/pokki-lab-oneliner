## 1. 이 프로젝트는 왜 시작됐는가

Pokki Lab은 포카마켓 안에서 빠르게 실험을 만들고, 검증하고, 종료할 수 있는 독립 실험 트랙을 만드는 작업이다.

핵심 배경은 아래와 같다.

- 기존 서비스 코드와 실험 코드를 강하게 분리하고 싶었다.
- 실험마다 인증, 웹뷰 브릿지, Supabase, 배포 구성을 반복해서 새로 만들고 싶지 않았다.
- 실험이 끝났을 때 레포, Vercel, Supabase까지 한 단위로 정리 가능한 구조가 필요했다.
- 첫 실험 후보인 `lab-concert-view`를 새 표준 위로 옮기기 전에, 그 기반이 되는 공용 템플릿이 먼저 필요했다.

이 배경에서 `pokki-lab-boilerplate`가 출발했다.

## 2. 프로젝트의 기본 방향

### 운영 원칙 — 초기 계획에서 확정된 방향

| 항목 | 방향 |
|------|------|
| 실험 단위 | `1 레포 = 1 실험 = 1 Vercel 프로젝트 = 1 Supabase 프로젝트` |
| 프레임워크 | Next.js App Router |
| 데이터 저장 | Supabase DB + Storage |
| 웹뷰 표준 | `#3545` 기반 인증/브릿지 패턴 |
| 템플릿 전략 | 공용 보일러플레이트를 GitHub Template처럼 사용 |
| 첫 적용 대상 | `lab-concert-view` |

### 프로젝트의 역할

이 레포는 실험 기능 자체를 담는 서비스 레포가 아니라, 이후 실험들이 출발할 공통 기반을 제공하는 템플릿 레포다.

즉 이 프로젝트의 목적은 새 실험을 위한 공통 문제를 먼저 해결해 두는 것이다.

- 공통 인증 처리
- 웹뷰 브릿지 액션
- 공통 라우팅 규칙
- Supabase 연결 패턴
- 디자인 토큰
- 시작/배포/셋업 문서

## 3. 처음 계획은 어떻게 잡혀 있었는가

기존 계획은 크게 두 산출물로 나뉘어 있었다.

### 산출물 A. `pokki-lab-boilerplate` 구축

실험을 새로 시작할 때 필요한 공통 구조를 별도 레포로 만든다.

포함하려던 범위:

- Next.js App Router 기본 골격
- 디자인 토큰
- `MainLayout`과 기본 UI 컴포넌트
- Supabase client/server 유틸
- 웹뷰 인증 유틸
- 토큰 refresh 흐름
- bridge action 호출 유틸
- `useWebViewRouter`
- CLAUDE 컨텍스트 및 작업용 문서

### 산출물 B. `lab-concert-view` 재구축

기존 `lab-concert-view`를 직접 조금씩 고치는 대신, 새 보일러플레이트 위로 도메인 자산만 다시 올리는 전략이었다.

핵심 포인트:

- 쿠키 기반 인증 흔적 제거
- `#3545` 기준 토큰 저장/refresh 패턴 적용
- bridge action / router 규칙 통일
- 실험 코드와 템플릿 코드를 분리

즉 원래 계획의 흐름은 아래와 같았다.

`보일러플레이트 구축 → 표준 정리 → lab-concert-view 재배치 → 이후 실험 확장`

## 4. 현재 코드베이스에는 무엇이 올라와 있는가

현재 레포를 기준으로 보면, 산출물 A에 해당하는 구조는 이미 상당 부분 구현되어 있다.

### 프로젝트 구조 — 현재 반영 상태

| 영역 | 현재 역할 | 상태 |
|------|----------|------|
| `app/` | App Router 진입점, 기본 페이지, 글로벌 스타일 연결 | 구현됨 |
| `app/_css/` | 색상/타이포/유틸/애니메이션 토큰 | 구현됨 |
| `components/Layout/` | 모바일 중심 메인 레이아웃 | 구현됨 |
| `components/ui/` | 기본 UI 컴포넌트 패턴 시작점 | 구현됨 |
| `webview/` | 토큰, refresh, fetcher, actions, router | 구현됨 |
| `lib/supabase/` | browser/server 클라이언트 유틸 | 구현됨 |
| `supabase/` | migration 예시, seed, config | 구현됨 |
| `docs/`, `scripts/` | 셋업/패턴/가이드 문서 | 구현됨 |
| `.claude/commands/` | 실험 작업용 문서형 커맨드 | 구현됨 |

### 현재 핵심 모듈

#### 웹뷰 인증/브릿지 영역

- `webview/token.ts`
  - `localStorage['accessToken']` 기준 저장/조회/삭제 담당
- `webview/refreshToken.ts`
  - 토큰 만료 시 네이티브 refresh 요청 담당
- `webview/fetcher.ts`
  - Authorization 헤더 자동 첨부
  - 401/403 시 refresh 후 1회 재시도
- `webview/platform.ts`
  - iOS/AOS 판별
- `webview/actions.ts`, `webview/action-types.ts`
  - 앱 브릿지 액션 매핑 및 실행
- `webview/useWebViewRouter.ts`
  - `os`/`token` 쿼리 보존 라우팅

#### 레이아웃/UI 영역

- `components/Layout/MainLayout.tsx`
  - 최대 425px 폭 기준의 모바일 중심 레이아웃
- `components/ui/button.tsx`
  - `cva + cn()` 패턴을 따르는 기본 버튼

#### Supabase 영역

- `lib/supabase/client.ts`
  - 브라우저 anon key 클라이언트
- `lib/supabase/server.ts`
  - server role 기반 서버 클라이언트
- `supabase/migrations/0001_lab_submissions.sql`
  - 범용 제출 테이블 예시

#### 운영 문서 영역

- `CLAUDE.md`
  - 이 레포의 작업 원칙과 구조 설명
- `README.md`
  - 템플릿 시작 가이드
- `docs/supabase-patterns.md`
  - DB/Storage 사용 패턴
- `scripts/setup-supabase.md`
  - Supabase 프로젝트 생성 및 연결 절차

## 5. 지금 이 프로젝트를 어떻게 이해하면 되는가

현재 상태를 한 문장으로 정리하면 이렇다.

`Pokki Lab의 공통 실험 템플릿은 이미 형태를 갖췄고, 지금은 그 기반을 실제로 반복 사용 가능한 수준으로 다듬는 단계다.`

즉 이 레포는 아직 실험 서비스 완성본은 아니지만, 방향성만 있는 문서 프로젝트도 아니다. 이미 공통 구조와 핵심 유틸이 코드로 올라와 있다.

### 현재 반영된 것

- 공통 폴더 구조
- 디자인 토큰 복사본
- 모바일 레이아웃
- 토큰 초기화 로직
- 웹뷰 토큰 저장/refresh/fetcher 패턴
- bridge action 호출 유틸
- Supabase 유틸과 migration 예시
- 문서/셋업 가이드

### 아직 다음 단계로 남아 있는 것

- 템플릿의 검증 경로를 더 안정적으로 정리
- 실제 실험 이관 시 바로 사용할 온보딩 문서 보강
- `lab-concert-view`를 별도 브랜치/레포에서 새 표준 위로 재배치

## 6. 기존 계획과 현재 구현의 관계

현재 레포는 전체 계획 중에서 `산출물 A`를 실체화하고 있는 결과물이다.

정리하면:

| 구분 | 의미 | 현재 상태 |
|------|------|-----------|
| 산출물 A | `pokki-lab-boilerplate` 구축 | 현재 이 레포에서 진행 중이며 대부분 골격 반영 |
| 산출물 B | `lab-concert-view` 재구축 | 아직 이 레포에서는 시작 전 |

중요한 점은, 현재 레포에 `lab-concert-view` 도메인 자산이 없는 것이 이상한 상태가 아니라는 점이다. 애초에 이 레포는 그 실험을 직접 담기 위한 공간이 아니라, 그 실험이 올라갈 기반을 만드는 공간이다.

## 7. 앞으로는 어떤 순서로 진행되는가

### 1단계. 보일러플레이트 정리

현재 만들어진 구조를 기준으로, 새 실험이 이 레포를 복제해서 바로 시작할 수 있게 마무리한다.

이 단계의 성격:

- 문서와 코드의 설명 일치
- 검증 절차 정리
- 템플릿 사용성 보강

### 2단계. `lab-concert-view` 이관

기존 실험을 새 표준으로 재배치한다.

이 단계에서 기대하는 변화:

- 인증 흐름 통일
- bridge/router 규칙 통일
- Pokki Lab 첫 실험의 표준화

### Phase 2. 실험 생성 규칙과 기획 시스템 고도화

Phase 1이 "실험을 시작할 수 있는 공통 템플릿"을 만드는 단계였다면, Phase 2는 "누가 시작해도 비슷한 품질과 구조로 실험이 만들어지게 하는 체계"를 만드는 단계다.

이 단계에서 보강하려는 축은 세 가지다.

#### 1. 코드 생성 규칙 고도화

단순히 템플릿 파일이 있는 수준을 넘어서, AI나 작업자가 새 페이지, 새 기능, 새 API, 새 테이블을 만들 때 항상 비슷한 결과물이 나오도록 프로젝트 전반의 룰셋을 더 강하게 고정하는 단계다.

특히 아래 영역이 중요하다.

- DB 연동 규칙
  - 어떤 경우에 migration을 추가하는지
  - 어떤 경우에 Route Handler를 만들고 어떤 경우에 클라이언트 읽기만 허용하는지
  - `user_id`를 어떤 방식으로 서버에서 결정하는지
- 사용자 인증 규칙
  - 토큰 사용 위치
  - `fetcher` 경유 원칙
  - refresh 실패 시 처리 방식
- 파일/폴더/네이밍 규칙
  - 페이지, 컴포넌트, API, migration의 생성 규칙

즉 Phase 2의 목표는 "프로젝트에 룰이 있다"가 아니라, "그 룰이 코드 생성 결과물까지 일관되게 강제된다"에 가깝다.

#### 2. PRD → SPEC 기획 에이전트 구축

새 실험은 단순히 화면 몇 개를 만드는 문제가 아니라, 왜 이 실험을 하는지, 어떤 지표를 보는지, 어떤 제약 아래 움직이는지까지 먼저 정리되어야 한다.

그래서 Phase 2에서는 신규 프로젝트 기획을 시작할 때, PRD를 읽고 아래 요소를 포함한 상세 SPEC으로 전개해주는 기획 에이전트가 필요하다.

- 프로젝트 목적과 실험 가설
- 유저 플로우
- 화면/기능 범위
- 데이터 모델과 저장 규칙
- 인증/권한 처리 방식
- 라우팅/브릿지 연동 포인트
- 예외 상황과 운영 규칙
- 구현 전 확인해야 할 컨벤션

이 에이전트의 역할은 단순 요약이 아니라, Pokki Lab의 목적성과 룰, 컨벤션, 플로우를 가진 실행 가능한 스펙을 만들어 주는 것이다.

#### 3. UI 공통 컴포넌트 셋업 규칙 정리

현재도 `MainLayout`, `Button`, 디자인 토큰, `cva + cn()` 패턴은 들어가 있지만, 앞으로는 "어떤 UI를 공통 컴포넌트로 승격할지"와 "새 UI를 어떤 기준으로 추가할지"를 더 분명히 정리할 필요가 있다.

예를 들면 아래 같은 기준이 해당된다.

- 공통 컴포넌트로 관리할 UI의 범위
  - 버튼, 인풋, 바텀시트, 다이얼로그, 탭, 카드, 폼 섹션 등
- 스타일링 규칙
  - 디자인 토큰 우선 사용
  - 임의 값 사용 허용 범위
  - variant/size 확장 방식
- 조합 규칙
  - `MainLayout` 위에서 어떻게 쌓는지
  - 모바일 웹뷰 기준 간격과 safe-area 처리 방식

즉 Phase 2의 UI 목표는 "공통 컴포넌트 몇 개가 있다"가 아니라, "실험 UI가 공통 언어 안에서 확장되도록 기준을 세운다"는 데 있다.

### 3단계. 후속 실험 확장

첫 이관이 끝나면, 이후부터는 새 실험이 이 보일러플레이트를 공통 출발점으로 삼게 된다.

이 시점부터 이 레포는 "개발 중인 템플릿"을 넘어 "운영 자산" 역할을 하게 된다.

## 8. 팀이 기억하면 되는 핵심 판단

### 현재 이 레포의 위치 — 요약

| 질문 | 답 |
|------|----|
| 이 레포는 무엇인가 | Pokki Lab 실험용 공통 보일러플레이트 |
| 왜 만들었는가 | 실험마다 반복되는 인증/브릿지/배포/Supabase 구성을 공통화하기 위해 |
| 지금 어느 단계인가 | 템플릿 기반은 대부분 구현됐고, 정리와 적용을 앞둔 단계 |

| 장기적으로 무엇이 되나 | 이후 실험들이 복제해서 쓰는 시작점 |

## 9. 결론

`pokki-lab-boilerplate`는 단순한 샘플 프로젝트가 아니라, Pokki Lab이라는 실험 운영 방식을 코드와 문서로 고정해 가는 기반 레포다.

기존 계획은 처음부터 `공용 템플릿 구축`과 `첫 실험 이관`의 두 축으로 잡혀 있었고, 현재 레포는 그중 첫 번째 축을 실제 코드로 옮기는 과정에 있다. 앞으로의 자연스러운 흐름은 이 기반을 정리한 뒤, `lab-concert-view`를 이 위로 올리고, 이후 실험들이 같은 구조를 반복 사용하게 만드는 것이다.
