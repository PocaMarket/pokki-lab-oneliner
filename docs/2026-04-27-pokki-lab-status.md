# Pokki Lab 프로젝트 현황 공유

작성일: 2026-04-27
기준 문서: `/Users/yein/.claude/plans/delightful-stargazing-chipmunk.md`
대상 레포: `pokki-lab-boilerplate`

## 1. 한 줄 요약

현재 이 레포는 `Pokki Lab`의 공용 보일러플레이트를 만드는 작업으로 보면 된다. 기준 플랜에서 정의한 `산출물 A: pokki-lab-boilerplate`는 대부분 골격이 올라와 있고, `산출물 B: lab-concert-view 재구축`은 아직 별도 레포/브랜치에서 시작하지 않은 상태다.

## 2. 프로젝트 목적

- 포카마켓 앱 안에서 빠르게 실험을 열고 닫을 수 있는 독립 실험 인프라를 만든다.
- 각 실험은 `1 레포 = 1 Vercel 프로젝트 = 1 Supabase 프로젝트` 원칙으로 운영한다.
- 공통 인증 규격은 `#3545`를 따른다.
- 첫 번째 적용 대상은 `lab-concert-view`이며, 기존 코드를 점진 보수하는 방식이 아니라 보일러플레이트 위로 다시 재배치하는 전략을 쓴다.

## 3. 현재 레포의 위치

이 레포는 `lab-concert-view` 자체가 아니라, 이후 실험들이 공통으로 복제해서 쓰게 될 템플릿 레포다. 즉 지금 팀이 해야 할 일은 크게 두 갈래다.

- `A. 보일러플레이트를 템플릿으로 안정화`
- `B. 그 다음 lab-concert-view를 새 표준 위로 이관`

## 4. 현재 상태 요약

### 완료되었거나 거의 완료된 항목

- Next.js 16 + React 19 + Tailwind v4 + Supabase 기반 기본 구조가 잡혀 있다.
- `webview/` 아래에 토큰 저장, refresh, fetcher, action bridge, router helper가 이미 들어가 있다.
- `CLAUDE.md`, `README.md`, `docs/supabase-patterns.md`, `scripts/setup-supabase.md` 등 운영 문서 초안이 존재한다.
- `.env.local.example`, `.eslintrc.json`, `.prettierrc`, Supabase migration 예시 파일까지 포함되어 있다.
- 메인 페이지는 최소한의 보일러플레이트 검증용 화면으로 구성되어 있다.
- `proxy.ts`를 통해 webview 여부를 헤더로 전달하는 처리도 들어가 있다.

### 아직 비어 있거나 마무리가 필요한 항목

- 실제 템플릿 품질을 보장할 검증 체인이 불안정하다.
- `lab-concert-view` 재구축 작업은 이 레포에서 아직 시작되지 않았다.
- 문서상 패턴은 정리되어 있지만, 실제 실험 이관 시 바로 참고할 수 있는 체크리스트성 문서가 부족하다.
- 예시 Route Handler나 end-to-end에 가까운 샘플 플로우는 아직 없다.

## 6. 지금 가장 중요한 우선순위

### P0. 보일러플레이트를 “복제 가능한 템플릿” 수준으로 고정

- 린트/빌드 검증 경로를 정상화해야 한다.
- 새 레포를 템플릿에서 생성했을 때, 환경변수만 넣고 바로 실행 가능한 상태인지 확인해야 한다.
- 문서와 실제 코드가 어긋나는 부분이 없는지 맞춰야 한다.

### P0. 인증/웹뷰 표준의 실제 동작 범위를 팀이 명확히 합의

- `#3545`에서 보일러플레이트가 책임지는 범위와 각 실험이 책임지는 범위를 분리해서 봐야 한다.
- 특히 `user_id` 추출을 템플레이트가 하지 않는다는 점을 팀 문서에서 더 강하게 명시할 필요가 있다.

### P1. `lab-concert-view` 이관 준비 문서화

- 재배치 브랜치 전략
- 보존 자산 목록
- 삭제 대상
- 인증 교체 포인트
- 검증 체크리스트

이 다섯 가지를 실행 순서대로 정리해두면 실제 이관 속도가 빨라진다.

### P2. 템플릿 운영성 강화

- CI 또는 최소 자동 검증 추가
- 예시 API/예시 업로드 흐름 추가
- 실험 시작용 체크리스트 정리

## 7. 확인된 이슈 / 리스크

### 1. `lint` 스크립트가 현재 정상 동작하지 않음

`2026-04-27` 기준 `npm run lint` 실행 시 `next lint`가 `Next 16` 환경과 맞지 않아 실패했다. 즉 템플릿의 기본 검증 명령어가 현재 깨져 있는 상태다.

영향:

- 템플릿 품질을 자동 확인하기 어렵다.
- 새 실험이 시작될 때 기본 신뢰도가 떨어진다.

권장 조치:

- `package.json`의 lint 명령을 현재 Next.js 버전에 맞는 방식으로 교체
- ESLint 단독 실행 기준으로 정리하는 방향이 현실적

### 2. `build` 검증은 샌드박스 환경에서 실패

`npm run build`는 Turbopack 내부 에러로 중단됐다. 오류 메시지는 CSS 처리 중 새 프로세스/포트 바인딩 권한 문제였고, 현재 실행 환경 제약의 영향 가능성이 있다.

해석:

- 이 실패를 제품 코드 버그로 바로 단정하긴 어렵다.
- 다만 "현재 이 레포가 문제없이 빌드된다"라고 말할 수 있는 상태도 아니다.

권장 조치:

- 로컬 일반 환경 또는 CI에서 재검증
- 필요하면 Turbopack/빌드 설정 우회 여부 확인

### 3. 실제 실험 이관을 검증할 샘플 플로우가 아직 없음

- 토큰 fallback
- refresh 재시도
- Supabase 쓰기 패턴
- bridge action 호출

이 네 가지는 문서만으로는 충분하지 않고, 최소 샘플 구현이나 smoke test가 있으면 훨씬 안전하다.
